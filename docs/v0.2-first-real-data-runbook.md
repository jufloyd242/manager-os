# v0.2 — First Real-Data Run Runbook

A step-by-step guide for moving from the sample fixture data to your real local
Obsidian vault and CSV exports. Follow the sections in order. Do not skip the
safety checks.

See [docs/v0.2-local-real-data-plan.md](v0.2-local-real-data-plan.md) for the
higher-level plan, acceptance criteria, and out-of-scope items.

---

## 2. Create Local `.env`

```bash
cp .env.example .env
```

Edit `.env` with real values:

```dotenv
# Absolute path to your real Obsidian vault
MANAGER_OS_VAULT_PATH=/absolute/path/to/your/vault

# Local DB path — stays in the gitignored data/processed/ directory
MANAGER_OS_DB_PATH=./data/processed/manager_os.duckdb

# CSV exports — place files in data/raw/ (gitignored) before setting these
MANAGER_OS_FORECAST_CSV=./data/raw/forecast.csv
MANAGER_OS_DEALS_CSV=./data/raw/deals.csv

# Daily workspace summary dir — leave default if you are not writing these yet
MANAGER_OS_WORKSPACE_SUMMARY_DIR=./data/raw/summaries

# Leave GWS blank — not in v0.2 scope
# MANAGER_OS_GWS_SNAPSHOT_DIR=./data/raw/gws_snapshots

# LLM — leave unset; the rule engine works without it
# OPENAI_API_KEY=sk-...
```

**Copy your CSV exports:**

```bash
cp /path/to/your/staffing-export.csv data/raw/forecast.csv
cp /path/to/your/deals-export.csv    data/raw/deals.csv
```

Confirm they are **not tracked**:

```bash
git ls-files data/raw/
# Must return only: data/raw/.gitkeep
```

## 3. Build Real Config Files

Edit these files in `config/`. The sample values are placeholders — replace them
with your actual team and clients. Do not leave the synthetic fixture names in
place; they will produce entity-resolution mismatches against your real notes.

### `config/people.yaml`

One entry per direct report (plus yourself if you appear in notes).

```yaml
- name: "Full Name"           # canonical — must match how you write it in notes
  aliases:
    - "FirstName"             # first-name-only variant
    - "firstname"
    - "first.last"            # email prefix if used in notes
    - "Full Name"
  role: "Senior ML Engineer"
  level: "L5"
```

**Alias tips:**
- Include every spelling variant you actually use in Obsidian (first name only,
  initials, Slack handle if written in notes).
- Aliases are matched case-insensitively, but list the natural spellings.
- The canonical `name` must match the `entity:` value in your note frontmatter
  (or be resolvable from it via an alias).
- Add yourself with an alias of `"I"` or `"me"` only if you use first-person
  attribution in action-item text.

### `config/clients.yaml`

One entry per active client/account.

```yaml
- name: "Client Org Name"     # canonical display name
  aliases:
    - "ClientOrg"
    - "client org"
    - "CON"                   # abbreviation you use in notes
  engagement: "Current Project Label"
```

**Alias tips:**
- Copy the exact strings you use in note frontmatter `entity:` fields and in
  note body text.
- If one org has multiple active engagements you write about separately, add
  the engagement name as an alias so both resolve to the canonical org.

### `config/deal_aliases.yaml`

Maps raw deal names from your CSV → canonical display names used in signals and
the brief.

```yaml
"Exact Deal Name From CSV": "Client — Engagement Label"
```

**How to populate:**
Open your deals CSV and copy every value from the deal name column verbatim.
The canonical right-hand side is what appears in signals, briefs, and the
dashboard. Unmapped deal names ingest fine and display as-is; add aliases
iteratively after the first run.

## 4. Safety Checks

Run these before writing anything to the real database.

### 4a. Readiness check

Run the built-in readiness command first. It validates env vars, source paths,
config files, and `.gitignore` rules in one pass:

```bash
manager-os readiness
# Exits 0 when no blocking failures (WARNs are allowed).
# Exits 1 when a FAIL is found — fix those before proceeding.
```

Resolve all FAILs before continuing. WARNs (e.g., sample config names still
present) are informational and do not block the run.

### 4b. Git state

```bash
git status --short
# Must be clean (no modified or untracked files beyond .env)
# .env itself must NOT appear in git status -- if it does, stop and check .gitignore

git ls-files data/ output/ .env
# Must return only: data/raw/.gitkeep
# If .env appears here, run: git rm --cached .env
```

### 4b. Confirm demo workflow still works after config changes

Your updated `people.yaml` and `clients.yaml` affect the demo pipeline too.
Confirm it still runs cleanly:

```bash
manager-os demo-reset --dry-run
# Should exit 0 and print the plan without errors
```

If `demo-reset --dry-run` fails after editing config, fix the config syntax
before proceeding.

### 4c. Profile your forecast CSV

```bash
manager-os profile-forecast
# Uses MANAGER_OS_FORECAST_CSV from .env by default.
# --path /path/to/other.csv to override.
# --json for machine-readable output.
```

The command shows:
- detected columns and their normalised internal names
- required fields found / missing
- unknown person or client names (compared to `config/people.yaml` / `config/clients.yaml`)
- overallocated rows (>100%), zero-allocation rows, missing or malformed dates

Exits 0 even with warnings. Exits 1 only if the file cannot be read or required
columns are completely absent.

If your export uses non-standard column headers, add them to
`forecast_column_aliases` in `config/source_priority.yaml` and re-run the
profiler to confirm they map correctly before ingesting.

### 4d. Profile your deals CSV

```bash
manager-os profile-deals
# Uses MANAGER_OS_DEALS_CSV from .env by default.
# --path /path/to/other.csv to override.
# --json for machine-readable output.
```

The command shows:
- detected columns and their normalised internal names
- required fields found / missing
- unknown client names (compared to `config/clients.yaml`)
- close dates within 14 days, missing/malformed close dates
- missing SOW/LOE status on deals with approaching close dates
- unassigned deals (no technical owner)
- late-stage deals with no staffing information

Exits 0 even with warnings. Exits 1 only if the file cannot be read or required
columns are completely absent.

If your export uses non-standard column headers, add them to
`deal_column_aliases` in `config/source_priority.yaml` and re-run the
profiler to confirm they map correctly before ingesting.

### 4e. Preview ingest and extract (dry run)

Use `--dry-run` to see what would be written without touching the database.

```bash
# Preview all sources
manager-os ingest --dry-run

# Preview a single source
manager-os ingest --dry-run --source forecast

# Preview extraction (reads existing notes, writes nothing)
manager-os extract --dry-run
```

`ingest --dry-run` scans each source and reports:
- how many files / rows were found per source
- how many are already in the database (would be skipped)
- warnings if a path is not configured or not found

`extract --dry-run` copies the existing notes to an in-memory database, runs
the rule engine there, and reports signal / action-item / decision counts —
without writing anything to the real database.

Both commands exit 0 and print
`⚠  Dry run — nothing was written to the database.`

Review the output. For each source:
- [ ] row counts look plausible (roughly match the CSV row count / vault file count)
- [ ] no "path not configured" or "not found" warnings
- [ ] `ingest --dry-run` shows 0 in "Would Skip" on first run (nothing in DB yet)
- [ ] `extract --dry-run` shows non-zero signal / action-item candidates

If anything looks wrong (missing columns, unknown paths), fix config and re-run.

### 4f. Spot-check with a temp database (optional)

If you want to inspect the normalised rows before writing to the real database,
ingest into a throwaway path first:

```bash
MANAGER_OS_DB_PATH=/tmp/manager_os_test.duckdb \
  manager-os ingest --verbose

MANAGER_OS_DB_PATH=/tmp/manager_os_test.duckdb \
  manager-os status
```

Then query the temp DB directly:

```bash
duckdb /tmp/manager_os_test.duckdb \
  "SELECT note_type, entity_name, note_date FROM notes ORDER BY note_date DESC LIMIT 10;"

duckdb /tmp/manager_os_test.duckdb \
  "SELECT person, week_start, allocation_pct FROM staffing_forecast ORDER BY week_start LIMIT 10;"

duckdb /tmp/manager_os_test.duckdb \
  "SELECT account, deal_name, sow_status, close_date FROM deals LIMIT 10;"
```

- [ ] `entity_name` in notes is non-empty and matches canonical names from
  `people.yaml` / `clients.yaml` (not raw alias variants)
- [ ] `person` in forecast rows matches canonical names
- [ ] `account` in deal rows matches canonical names

If entity names are blank or wrong, update `aliases` in the relevant config
file and re-run against the temp DB.

## 6. Review Criteria

After Step 5, read `output/daily_briefs/YYYY-MM-DD.md` and score it.

### Per-signal review

For each surfaced signal, record your assessment:

| Question | Target |
|----------|--------|
| Do I already know about this? | OK on day 1; should decrease over time |
| Is this actionable today? | Yes → useful signal |
| Does `why_it_matters` give enough context? | Should reference entity and source |
| Can I find the source note/row without guessing? | Each signal stores `source_path` |
| Is the entity name correct? | Wrong name → alias gap in config |

**Signal types fired by the rule engine:**

| Signal type | Condition | Severity |
|-------------|-----------|----------|
| `risk` | Note body contains a risk keyword¹ | high |
| `people_health` | No 1:1 note for a person in the last 14 days | medium |
| `sow_loe_review` | Deal closing within 7 days with unsigned SOW | high |
| `utilization_risk` | Team member at >100% allocation in the next 14 days | high |

¹ Risk keywords: `at risk`, `escalat`, `bloated`, `delay`, `delayed`, `miss`,
`missed`, `blocked`, `blocking`, `concern`, `concerned`, `overdue`, `urgent`,
`critical`, `red flag`

### Brief quality checklist

- [ ] At least one signal I did not already have in my head
- [ ] No signal with a clearly wrong entity name
- [ ] Open action items include real commitments from recent 1:1 notes
- [ ] Meeting prep (if run) covers today's actual meetings
- [ ] Brief is readable in under 5 minutes

### Acceptance threshold

v0.2 passes when, over 3 consecutive real-data runs:
- 60–70% of surfaced signals are marked useful
- `failed = 0` on all ingest sources
- No real files appear in `git status` or `git ls-files`

## 7. Feedback Loop

Use this process after each daily run to improve signal quality.

### Rate signals

After reading the brief, use the CLI to record your assessment of each signal:

```bash
# Rate a signal by its ID (shown in the DB or via status command)
manager-os signal-rate <SIGNAL_ID> useful
manager-os signal-rate <SIGNAL_ID> not_useful --note "Already knew this"
manager-os signal-rate <SIGNAL_ID> duplicate --note "Same as risk from yesterday"
manager-os signal-rate <SIGNAL_ID> wrong_entity --note "Should be Alice, not Bob"
manager-os signal-rate <SIGNAL_ID> too_low_priority
manager-os signal-rate <SIGNAL_ID> snoozed --snooze-until 2026-06-21
manager-os signal-rate <SIGNAL_ID> resolved

# View the aggregate usefulness report
manager-os signal-feedback
```

**Supported ratings:**

| Rating | Meaning |
|--------|---------|
| `useful` | Actionable, correct entity, worth knowing |
| `not_useful` | Noise — already knew it, stale, or irrelevant |
| `duplicate` | Already surfaced by another signal |
| `wrong_entity` | Signal fired on the wrong person/client/deal |
| `too_low_priority` | Real issue but not worth surfacing right now |
| `snoozed` | Temporarily silence until `--snooze-until` date |
| `resolved` | Underlying issue is already handled |

Signals rated `not_useful`, `duplicate`, `wrong_entity`, or `resolved` are
automatically suppressed from the next brief. Snoozed and too_low_priority
signals remain visible so you can track them.

To find signal IDs, query the DB or use `manager-os status`:

```bash
duckdb data/processed/manager_os.duckdb \
  "SELECT id, signal_type, severity, entity_name, summary
   FROM signals
   WHERE signal_date = CURRENT_DATE AND rating IS NULL
   ORDER BY severity, signal_type;"
```

### Capture wrong entities

If `entity_name` is wrong or blank:

```bash
# Find notes where entity_name is empty
duckdb data/processed/manager_os.duckdb \
  "SELECT note_date, note_type, source_path FROM notes WHERE entity_name = '' OR entity_name IS NULL LIMIT 20;"
```

Fix: add the missing alias to `config/people.yaml` or `config/clients.yaml`,
then re-ingest with `--force` to reprocess the affected files:

```bash
manager-os ingest --source obsidian --force --verbose
```

### Capture missing signal types

If something you care about isn't surfacing, note it. Candidate additions for
v0.3:
- Deal stage stalled (no stage change in N days)
- Client with no note in N days (client health cadence)
- Action item overdue (committed by date passed without `closed_at`)
- Person with open action item count above threshold

Record these observations in a `docs/signal-backlog.md` (not tracked; your
personal notes) for v0.3 scoping.

### Iterative config tuning

After each run where entity resolution is wrong, update the relevant config
file and re-ingest from the affected source. Changes are picked up on the next
run automatically — no code changes required.

## 8. Rollback / Reset

### Option A — Delete the real DB and rebuild from scratch

All ingest is idempotent. Re-running on the same source files produces an
identical database.

```bash
rm data/processed/manager_os.duckdb
manager-os ingest --date $(date +%Y-%m-%d) --verbose
```

### Option B — Delete only derived data, keep raw documents

The `notes`, `signals`, `action_items`, `decisions`, `daily_briefs`, and
`meeting_prep` tables are all derived from `raw_documents`. If you want to
re-extract with updated config:

```bash
duckdb data/processed/manager_os.duckdb "
  DELETE FROM signals;
  DELETE FROM action_items;
  DELETE FROM decisions;
  DELETE FROM daily_briefs;
  DELETE FROM meeting_prep;
"
manager-os extract --verbose
manager-os brief
```

### Option C — Return to demo mode

Demo reset only touches `data/demo/manager_os_demo.duckdb` and `output/demo/`.
It does not affect your real-data DB.

```bash
manager-os demo-reset --dry-run   # preview
manager-os demo-reset             # run
```

### Option D — Delete generated output only

```bash
rm -rf output/daily_briefs/ output/meeting_prep/ output/closeout/
# Regenerate:
manager-os brief && manager-os meeting-prep && manager-os closeout
```

### Never do this

- Do **not** delete your real Obsidian vault or CSV source files.
- Do **not** run `manager-os demo-reset` thinking it resets your real DB — it
  only touches the demo DB path (`data/demo/`).
- Do **not** `git add data/` — check `git status` before every commit.

## Appendix: Obsidian Frontmatter Reference

For notes to ingest correctly, add a YAML frontmatter block. Minimum fields:

```yaml
---
type: 1on1          # or: client, deal, meeting, team, practice
entity: Full Name   # canonical name or an alias from people.yaml / clients.yaml
date: 2026-06-14
---
```

**Recognized `type` values:**
`1on1`, `one-on-one`, `client`, `client-status`, `deal`, `meeting`, `team`,
`practice`

**Fallback — directory name:** If a note has no `type:` field, the ingestor
checks the parent directory name. Recognized directory names:
`1on1/`, `1-on-1/`, `one-on-ones/`, `clients/`, `client-notes/`, `deals/`,
`deal-notes/`, `meetings/`, `team/`, `practice/`

Notes without a recognized type still ingest as `raw_documents` but will not
contribute to signals or the brief until a type is inferred.
