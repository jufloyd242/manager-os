# v0.2 — Local Real Data Plan

**Goal:** Move Manager OS from synthetic fixture data to real local work data
so it produces a trusted daily management command center that answers:

1. What needs Justin's attention today?
2. What changed since the last run?
3. What decisions/actions does Justin owe?
4. What risks are forming or worsening?
5. What meetings need prep?

**Scope:** Real Obsidian vault + real CSV exports only.
No Gmail, no Chat, no Drive live sync, no write-back, no cloud model required.

---

## 1. Safety Checklist (before connecting anything)

Run through this list once before setting any real path in `.env`.

- [ ] Confirm `.env` is in `.gitignore` and is not tracked: `git ls-files .env`
  → must return empty.
- [ ] Confirm `data/` and `output/` are excluded: `git ls-files data/ output/`
  → must return only `data/raw/.gitkeep`.
- [ ] Confirm no `*.duckdb` files are tracked: `git ls-files '*.duckdb'`
  → must return empty.
- [ ] Back up the demo database before the first real run:
  `cp data/demo/manager_os_demo.duckdb data/demo/manager_os_demo.duckdb.bak`
- [ ] Choose a dedicated DB path for real data that is **not** inside the repo,
  or confirm that `data/processed/` is gitignored (it is).
  Recommended: `./data/processed/manager_os.duckdb` (default, gitignored).
- [ ] Export the real forecast CSV and deals CSV to `data/raw/` (gitignored).
  Do not copy them anywhere that is tracked.
- [ ] Confirm your Obsidian vault path is outside the repo root.
  `MANAGER_OS_VAULT_PATH` should be an absolute path.
- [ ] Run `manager-os demo-reset --dry-run` once more to confirm demo paths
  still resolve to fixture directories, not your real vault.

## 2. Required `.env` Values

Minimum set for a real-data v0.2 run. Copy `.env.example` to `.env` and fill in:

```dotenv
# Absolute path to your real Obsidian vault
MANAGER_OS_VAULT_PATH=/absolute/path/to/your/vault

# Local DB — stays in the gitignored data/processed/ directory
MANAGER_OS_DB_PATH=./data/processed/manager_os.duckdb

# Exported CSV files — place in data/raw/ (gitignored)
MANAGER_OS_FORECAST_CSV=./data/raw/forecast.csv
MANAGER_OS_DEALS_CSV=./data/raw/deals.csv

# Daily workspace summary markdown files (optional; skip if not writing these)
MANAGER_OS_WORKSPACE_SUMMARY_DIR=./data/raw/summaries

# Leave GWS blank / default — not in v0.2 scope
# MANAGER_OS_GWS_SNAPSHOT_DIR=./data/raw/gws_snapshots
```

Do not set `OPENAI_API_KEY` or `GEMINI_MODEL` unless you want LLM extraction.
The rule engine produces useful signals without any model.

## 3. Config Files That Need Real Values

### `config/people.yaml`

Replace the five synthetic team members with your actual direct reports.
Minimum fields per entry: `name`, `aliases`, `role`.

```yaml
- name: "Jane Smith"            # canonical display name (match how you write it in notes)
  aliases:
    - "Jane"
    - "jane"
    - "jane.smith"
    - "Jane Smith"
  role: "Senior ML Engineer"
  level: "L5"
```

**Tips:**
- The `aliases` list drives entity resolution. Include every spelling variant
  you actually use in notes (first name only, email prefix, Slack handle if
  you write it in notes).
- The canonical `name` must match exactly what appears in your Obsidian
  frontmatter `entity:` field, or be resolvable via an alias.
- Add an entry for yourself if you appear in notes (e.g., commitments attributed
  to you).

### `config/clients.yaml`

Replace with your actual client organizations.

```yaml
- name: "Client Org Name"       # canonical
  aliases:
    - "ClientOrg"
    - "client org"
    - "CON"                     # abbreviation you use
  engagement: "Project Name"    # current primary engagement label
```

**Tips:**
- Aliases are case-insensitive at match time, but list the spellings you
  actually use.
- If a single client has multiple active engagements, add them as separate
  alias variants or use the engagement name as an alias.

### `config/deal_aliases.yaml`

Maps raw deal names as they appear in your deals CSV → canonical display names.

```yaml
"Raw Deal Name From CSV": "Client — Engagement Label"
```

**Tips:**
- Open your deals CSV and copy the exact strings from the deal name column.
- The canonical name is what will appear in signals, briefs, and dashboard.
- Unmapped deal names still ingest — they just display as-is. Add aliases
  iteratively after the first run.

### `config/source_priority.yaml`

Review the `forecast_column_aliases` and `deal_column_aliases` sections.
If your CSV exports use non-standard column headers, add them here.
The `confidence_weights` and `conflict_resolution_order` require no changes
for a basic v0.2 run.

## 4. Validating Obsidian Ingestion

Goal: confirm notes ingest correctly without committing any note content.

```bash
# Dry-run: see what the ingestor finds without writing to DB
manager-os ingest --source obsidian --date $(date +%Y-%m-%d) --verbose

# If DB is new, check the raw_documents and notes tables
manager-os status
```

**Checklist:**
- [ ] `ingested` count is non-zero.
- [ ] `failed` count is 0 (check `--verbose` output for error messages).
- [ ] `manager-os status` shows note count > 0 and `Latest note date` is recent.
- [ ] Spot-check one note: connect to the DB and query
  `SELECT note_type, entity_name, note_date FROM notes LIMIT 10;`
  using `duckdb data/processed/manager_os.duckdb`.
  Verify `note_type` is being inferred (1on1, client, deal, meeting) and
  `entity_name` resolves to a canonical name from `people.yaml` or
  `clients.yaml`.

**Common issues:**
| Symptom | Likely cause | Fix |
|---------|-------------|-----|
| `note_type = ""` for all notes | Frontmatter missing `type:` field and dir name not recognized | Add `type: 1on1` (or `client`, `deal`) to note frontmatter, or rename parent directories to match `_DIR_TYPE_MAP` (`1on1/`, `clients/`, `deals/`, `meetings/`) |
| `entity_name = ""` | Frontmatter missing `entity:` field | Add `entity: Person Name` to frontmatter |
| Entity not resolving to canonical name | Alias not in `people.yaml` or `clients.yaml` | Add the variant spelling as an alias |
| `failed` > 0 | Malformed frontmatter YAML | Fix the note or add it to a skip list |

## 7. First Real-Data Run Workflow

Run each step individually, verify, then proceed.

```bash
# Step 0 — confirm demo DB still works after config changes
manager-os demo-reset --dry-run

# Step 1 — seed people and clients from updated config, ingest all sources
manager-os ingest --date $(date +%Y-%m-%d) --verbose

# Step 2 — review ingest results; fix any 0-count or high-failure sources
manager-os status

# Step 3 — extract signals, action items, decisions
manager-os extract --date $(date +%Y-%m-%d) --verbose

# Step 4 — review what signals were generated
#   (DuckDB CLI or status command)
manager-os status

# Step 5 — generate the brief
manager-os brief --date $(date +%Y-%m-%d)

# Step 6 — read the brief
#   output/daily_briefs/YYYY-MM-DD.md

# Step 7 — optionally generate meeting prep
manager-os meeting-prep --date $(date +%Y-%m-%d)

# Step 8 — at end of day
manager-os closeout --date $(date +%Y-%m-%d)
```

**After each ingest step:** run `manager-os status` and confirm counts are
growing. If a table stays at 0 after its source is ingested, stop and diagnose
before proceeding.

## 8. Rollback / Reset Procedure

### Option A — Delete the real DB and start over

```bash
rm data/processed/manager_os.duckdb
manager-os ingest --date $(date +%Y-%m-%d)
```

All ingest is idempotent. Re-running on the same source files rebuilds the DB
in the same state. The content-hash dedup guarantees no phantom duplicates.

### Option B — Reset to demo mode

```bash
manager-os demo-reset --date $(date +%Y-%m-%d)
```

This touches only `data/demo/manager_os_demo.duckdb` and `output/demo/`.
It does not affect your real-data DB.

### Option C — Confirm nothing real was committed

```bash
git status --short
git ls-files data/ output/
# Both should show nothing except data/raw/.gitkeep
```

## 9. Judging Whether the Brief Is Actually Useful

Review the generated `output/daily_briefs/YYYY-MM-DD.md` against these
criteria after the first few real-data runs.

### Signal usefulness rubric

For each surfaced signal, ask:

| Question | Target |
|----------|--------|
| Do I already know about this? | Expected for the first run — signals should stabilize |
| Is this something I actually need to act on? | Yes → signal is useful; No → may indicate a noisy rule |
| Does it tell me **why** it matters? | The `why_it_matters` field should give context |
| Can I trace it to a source note or CSV row? | Each signal stores `source_path` |

### Noise vs. signal

The rule engine currently fires on:
- **Risk keywords** in note bodies: `at risk`, `escalat`, `blocker`, `delay`,
  `delayed`, `miss`, `missed`, `blocked`, `blocking`, `concern`, `concerned`,
  `overdue`, `urgent`, `critical`, `red flag`
- **Stale 1:1**: no 1:1 note for a person in the last 14 days
- **SOW deadline**: deal closing within 7 days with unsigned SOW
- **Overallocation**: team member at >100% in the next 14 days

If risk-keyword signals are too noisy (firing on historical notes you don't
care about), note the source paths. The fix in v0.3 would be to weight signals
by recency or add a note-age filter.

### Useful brief checklist

- [ ] At least one signal I didn't already have in my head
- [ ] No signal with an obviously wrong entity name (indicates alias gap)
- [ ] Open action items include real commitments I made (check note 1:1 sections)
- [ ] Meeting prep section covers today's actual meetings
- [ ] Brief is readable in under 5 minutes

## 10. Acceptance Criteria for v0.2

v0.2 is complete when all of the following are true:

| Criterion | How to verify |
|-----------|--------------|
| Real Obsidian notes ingest without errors | `manager-os ingest --source obsidian --verbose` → `failed = 0` |
| Real forecast CSV ingests with correct person names | `staffing_forecast` rows have non-empty `person` values matching `people.yaml` canonical names |
| Real deals CSV ingests with correct account names | `deals` rows have non-empty `account` values; SOW signals fire for qualifying deals |
| Daily brief generates in < 10 seconds | Time `manager-os brief` |
| Brief is readable in < 5 minutes | Manual review |
| At least 60–70% of surfaced signals are useful | Manual signal review: mark each as useful / noise after first 3 real-data runs |
| Every signal has `source_path` | `SELECT count(*) FROM signals WHERE source_path IS NULL OR source_path = '';` → 0 |
| No real notes, CSVs, or DB files committed | `git ls-files data/ output/ '*.duckdb'` → empty except `.gitkeep` |
| Demo workflow still passes after config changes | `manager-os demo-reset --dry-run` exits 0; `pytest tests/ -q` → 426 passed |
